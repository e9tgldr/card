# Guest accounts

**Date:** 2026-04-30
**Status:** Design (pending user review)

## Goal

A signed-in user (the **parent**) may spawn up to 5 **guest** auth accounts so friends without access codes can sign into them and play multiplayer with the parent. Guests are real `auth.users` rows with their own `profiles` and `active_sessions` records, so they appear as distinct identities in live rooms. All EXP earned by guests rolls up to the parent on the public leaderboard. Guests cannot host live rooms, redeem access codes, or claim cards.

## Non-goals

- Multi-tier hierarchies. Guests cannot themselves become parents.
- Self-service guest signup. The only way to obtain a guest account is via a link generated by a parent.
- Cross-parent guest reassignment. A guest's `parent_user_id` is immutable; rotating a slot to a different friend reuses the same `auth.users` row but the linkage to the parent never changes.
- Public visibility of guest identities on the leaderboard. Guest usernames never appear in any public ranking; only the parent's name with the rolled-up score.
- Guest card collection. Guests do not own cards (`card_ownership` rows). They can only join rooms hosted by parents who own the cards in question.
- Email-based magic links. Guests have synthetic `@guests.local` emails; the "magic link" is a parent-shared URL, not an inbox.

## Behavior

### Parent initialises slots

1. Parent visits `/profile/guests` (or opens the inline `<GuestSlotsPanel/>` from the live-room create modal) for the first time.
2. Client calls `guest-init-slots`. The edge fn idempotently creates 5 `guest_slots` rows for the parent if they don't already exist, each with `slot_idx = 1..5`, `auth_user_id = NULL`, `claimed_at = NULL`. No `auth.users` rows are created yet.
3. Panel renders 5 empty slot cards.

### Parent generates a join link for a slot

1. Parent clicks **Generate link** on slot N.
2. Client calls `guest-generate-token({ slot_idx: N })`. The edge fn:
   - verifies caller is the parent of that slot (and is not themselves a guest),
   - rate-limits: rejects with 429 if the parent has generated >10 tokens in the last hour,
   - if the slot has no `auth_user_id` yet, creates the synthetic guest user (see *Guest user creation* below) and links it to the slot,
   - generates a 32-byte raw token, stores `sha256(token)` in `guest_tokens` with `expires_at = now() + 15 minutes`, single-use, `slot_id` linked,
   - returns the raw token (only shown once) plus the full URL `https://<host>/guest/join?token=<raw>`.
3. UI shows a one-shot copy box with the URL and a 15-minute countdown. Closing/regenerating invalidates the previous token (DELETE on the prior `guest_tokens` row for that slot).

### Friend claims a link

1. Friend opens the URL on their device.
2. The `/guest/join` page emits a `<meta name="referrer" content="no-referrer">` and (if the platform allows) sends a same-origin `Referrer-Policy: no-referrer` header to suppress token leakage via the Referer header.
3. The page reads `?token=` from `location`, then immediately calls `history.replaceState(null, '', '/guest/join')` to scrub the token from the address bar and browser history.
4. Page calls `guest-claim-token({ token })`. The edge fn:
   - hashes the token, looks up the row with `FOR UPDATE`. If missing or `expires_at < now()` → 410 with generic `expired_or_invalid` reason (no enumeration leak),
   - DELETEs the token row inside the same transaction (atomic single-use; loses race → 409),
   - generates a fresh password for the guest user, calls `auth.admin.updateUserById(auth_user_id, { password })`,
   - calls `auth.admin.generateLink({ type: 'magiclink', email })` — but instead of emailing it, returns a freshly issued access+refresh token pair to the client by calling `auth.admin.createSession` (Supabase admin API). Client receives `{access_token, refresh_token}` and does `supabase.auth.setSession(...)`,
   - calls `claim_session_atomic(p_user_id=guest_id, p_session_id=<new uuid>, p_device_label='guest:<friend label>', p_force=true, p_stale_seconds=120)` — the existing single-device claim function. `force=true` evicts any prior friend on the same slot,
   - returns `{ok: true, parent_username, guest_username, session_id}`.
5. Client stores `session_id`, starts the existing 30s heartbeat, and routes to `/`.

### Guest plays

- Live-room **Create Room** button is hidden in the UI; if the friend manually hits `game-create-session` for `live_room` or `tournament` mode → 403.
- Redeem-code page UI is hidden; if hit directly → `redeem-code` returns 403.
- Card-claim flows are hidden; the `/figures` browse and `/order` shop are visible read-only but purchase/claim actions are disabled.
- An in-game banner reads (mn): `+{n} XP → {parentDisplay}` whenever the guest gains score, sourced from `profiles.parent_user_id → profiles.username`. The banner shows in-room and on the post-round summary; it is suppressed on any view rendered to other users (spectator, public replay).
- Tournament submissions: `game_results` insert trigger sets `tournament_owner_id = coalesce(p.parent_user_id, p.id)`. The partial unique index `(tournament_owner_id, tournament_id) WHERE tournament_id IS NOT NULL` lets the **first** member of a parent+guests "family" submit, and rejects every subsequent family member's submission for that tournament with a unique-violation. The client surfaces this as "Someone in your group already submitted to this tournament."
- Every game-mutating edge fn (`game-create-session`, `game-submit-result`, `game-live-event`, `game-live-snapshot`) calls a new shared helper `assertActiveSession(userJwt, providedSessionId)` that checks `active_sessions.session_id` matches what the client claims. If not → 401. This closes the JWT-after-revoke gap (Codex fix) and is the same defense the existing single-device feature was missing.

### Parent revokes a slot

1. Parent clicks **Revoke** on slot N from `<GuestSlotsPanel/>`.
2. Client calls `guest-revoke-slot({ slot_idx: N })`. The edge fn:
   - verifies caller owns that slot,
   - DELETEs the slot's `active_sessions` row (kills heartbeat continuity),
   - rotates the guest's password via `auth.admin.updateUserById` (forces a fresh sign-in next time),
   - calls `auth.admin.signOut(guest_id, 'global')` (revokes refresh tokens — access tokens still valid until exp, but `assertActiveSession` will reject them on next mutation),
   - DELETEs any pending `guest_tokens` rows for that slot.
3. The slot card flips back to "empty + reassignable". The previous friend's device sees the next heartbeat / next mutation return 401, fires the existing eviction toast, and routes to `/`.

### Parent regenerates a link for an already-claimed slot (rotation)

Same flow as **generate** above; the existing guest's `auth.users` row is preserved (so historical `game_results.tournament_owner_id` rollups remain coherent), but the password is rotated and `active_sessions` is cleared. The new friend who claims it ends up signed in as the same underlying auth user with a fresh password.

### Parent hits the cap

`guest-generate-token` returns 409 if the parent has already generated 5 distinct guest users (one per slot) and is trying to spawn a 6th. The cap is per-parent, enforced both by the `CHECK (slot_idx BETWEEN 1 AND 5)` on `guest_slots` and the trigger that only allows up to 5 `guest_slots` rows per parent.

### Guest user creation (deterministic synthetic email)

When a slot is first activated:

```
email     = `guest_${parent_uuid_first_12}_${slot_idx}@guests.local`
username  = `guest_${parent_uuid_first_12}_${slot_idx}`  (pre-claim placeholder)
password  = crypto.randomUUID()  (rotated on every claim)
```

After first claim, the friend can change `display_name` to whatever (subject to length + profanity validation). The deterministic email guarantees no collision regardless of parent username changes; the `_first_12` truncation keeps it readable while preserving uniqueness (collision probability over a small parent pool is negligible).

returns trigger language plpgsql as $$
begin
  if old.parent_user_id is distinct from new.parent_user_id then
    raise exception 'parent_user_id is immutable';
  end if;
  return new;
end $$;

returns trigger language plpgsql as $$

  if new.parent_user_id is not null then
    if new.parent_user_id = new.id then

      select 1 from public.profiles p

-- so a guest who happens to be the family's first submitter still credits the
-- parent on the medal table.
create or replace function public.finalize_tournament(tid uuid)
returns void
language plpgsql
security definer
as $$
declare r record;
begin
  for r in
    select tournament_owner_id as winner_id, row_number() over (
      order by score desc, completed_at asc, tournament_owner_id
    ) as rn
    from public.game_results
    where tournament_id = tid
    order by score desc, completed_at asc, tournament_owner_id
    limit 3
  loop
    insert into public.user_achievements (user_id, kind, ref_id)
    values (
      r.winner_id,
      case r.rn when 1 then 'tournament_gold'
                when 2 then 'tournament_silver'
                else 'tournament_bronze' end,
      tid
    )
    on conflict (user_id, ref_id)
      where kind in ('tournament_gold','tournament_silver','tournament_bronze')
      do nothing;
  end loop;

  update public.tournaments set published = true where id = tid and not published;
end;
$$;

-- 8. claim_guest_token_atomic: single round-trip atomic single-use claim.
create or replace function public.claim_guest_token_atomic(p_token_hash bytea)
returns table (
  found            boolean,
  slot_id          uuid,
  auth_user_id     uuid,
  parent_username  text,
  guest_username   text
)
language plpgsql
security definer
set search_path = public
as $$
declare v_row record;
begin
  select t.id as token_id,
         s.id as slot_id,
         s.auth_user_id,
         pp.username as parent_username,
         gp.username as guest_username
    into v_row
  from public.guest_tokens t
  join public.guest_slots  s  on s.id = t.slot_id
  join public.profiles     pp on pp.id = s.parent_user_id
  left join public.profiles gp on gp.id = s.auth_user_id
  where t.token_hash = p_token_hash and t.expires_at > now()
  for update of t;

  if not found then
    return query select false, null::uuid, null::uuid, null::text, null::text;
    return;
  end if;

  delete from public.guest_tokens where id = v_row.token_id;

  return query
    select true, v_row.slot_id, v_row.auth_user_id, v_row.parent_username, v_row.guest_username;
end;
$$;

revoke all on function public.claim_guest_token_atomic(bytea) from public;
grant execute on function public.claim_guest_token_atomic(bytea) to service_role;

create or replace function public.select_profile_with_parent(p_user_id uuid)
returns table (
  id              uuid,
  username        text,
  is_admin        boolean,
  parent_user_id  uuid,
  parent_username text
)
language sql

as $$
  select p.id, p.username, p.is_admin, p.parent_user_id, pp.username
  from public.profiles p
  left join public.profiles pp on pp.id = p.parent_user_id
  where p.id = p_user_id and (p.id = auth.uid() or auth.role() = 'service_role');
$$;

grant execute on public.select_profile_with_parent(uuid) to authenticated, service_role;

## Edge functions

All deployed via the Supabase MCP tool, following the existing `_shared/cors.ts` + service-role pattern. JWT verification against `auth.uid()` always derives `parent_user_id` from `profiles` server-side — never trust JWT/user metadata.

### New shared helper: `_shared/assertActiveSession.ts`

```ts
export async function assertActiveSession(
  sb: SupabaseClient,
  userId: string,
  providedSessionId: string,
): Promise<void> {
  const { data, error } = await sb
    .from('active_sessions')
    .select('session_id, last_seen')
    .eq('user_id', userId)
    .single();
  if (error || !data || data.session_id !== providedSessionId) {
    throw new HttpError(401, 'session_revoked');
  }
}
```

Called by `game-create-session`, `game-submit-result`, `game-live-event`, `game-live-snapshot`. Admins bypass via the existing `is_admin` check (same pattern as `claim-session`). The client passes `session_id` in a custom header `X-Session-Id` on every game-mutating request. This same helper closes the existing JWT-after-revoke gap in single-device sign-in (defense in depth).

### `guest-init-slots`

**Request** (`POST`, no body): authenticated as parent.

**Logic:**

1. Verify JWT → `user_id`. Reject 401 if missing.
2. Read `profiles.parent_user_id` for `user_id`. If non-null → 403 `guests cannot init slots`.
3. UPSERT 5 rows into `guest_slots` for `parent_user_id = user_id` with `slot_idx = 1..5`, `ON CONFLICT (parent_user_id, slot_idx) DO NOTHING`.
4. Return `{ok: true}`.

Idempotent: safe to call on every panel mount.

### `guest-generate-token`

**Request:** `POST { slot_idx: 1..5 }`.

**Logic:**

1. Verify JWT → `user_id`. Reject if guest.
2. Rate-limit: count `guest_tokens` rows for this parent in the last hour via `slot_id IN (select id from guest_slots where parent_user_id = $1)`. If ≥10 → 429.
3. Look up the slot row. If absent → 404.
4. If `slot.auth_user_id IS NULL`:
   - synthesize `email`, `username`, random `password` per the deterministic scheme,
   - `auth.admin.createUser({ email, password, email_confirm: true })` → `new_user_id`,
   - INSERT `profiles { id: new_user_id, username, parent_user_id: user_id }`,
   - UPDATE the slot row with `auth_user_id = new_user_id`.
5. Generate `raw_token = crypto.randomBytes(32)` and `token_hash = sha256(raw_token)`. INSERT into `guest_tokens` with `expires_at = now() + 15min`, deleting any prior unexpired token for the same slot.
6. Return `{ok: true, url: 'https://<host>/guest/join?token=<base64url(raw_token)>', expires_at}`.

### `guest-claim-token`

**Request:** `POST { token: '<base64url>' }`. **No** Authorization header (claimed unauthenticated).

**Logic:**

1. `token_hash = sha256(base64url_decode(token))`.
2. Inside an RPC `claim_guest_token_atomic(p_token_hash bytea)` (service-definer, `FOR UPDATE` lock, single round trip):
   - SELECT the row + slot + parent username FOR UPDATE,
   - if missing or expired → return `{found: false}`,
   - DELETE the token row,
   - return `{found: true, slot_id, auth_user_id, parent_username}`.
3. If `found: false` → 410 `expired_or_invalid` (single generic code, no enumeration leak).
4. Generate a new `password = crypto.randomUUID()`. `auth.admin.updateUserById(auth_user_id, { password })`.
5. Call `auth.admin.signOut(auth_user_id, 'global')` to revoke any prior friend's refresh tokens.
6. Get a fresh session: `auth.admin.generateLink({ type: 'magiclink', email })` returns an `action_link` containing a `?token=...` we can resolve server-side via `auth.verifyOtp({ type: 'magiclink', token_hash })`. Use the resulting `{access_token, refresh_token}` — return them to the client.
7. UPDATE `guest_slots.claimed_at = now()`.
8. Generate `new_session_id = crypto.randomUUID()`. Call `claim_session_atomic(auth_user_id, new_session_id, 'guest', force=true, stale_seconds=120)`.
9. Return `{ok: true, access_token, refresh_token, session_id: new_session_id, parent_username, guest_username}`.

If any step after (3) throws, the token row is already gone (single-use is preserved), but the friend can ask the parent for a new link.

### `guest-revoke-slot`

**Request:** `POST { slot_idx: 1..5 }`.

**Logic:**

1. Verify JWT → `user_id`. Reject if guest.
2. Look up the slot. If absent → 404. If `auth_user_id IS NULL` → no-op, return `{ok: true}`.
3. DELETE `active_sessions WHERE user_id = slot.auth_user_id` (kills heartbeat continuity).
4. `auth.admin.updateUserById(slot.auth_user_id, { password: crypto.randomUUID() })` (forces fresh sign-in next claim).
5. `auth.admin.signOut(slot.auth_user_id, 'global')` (revokes refresh tokens).
6. DELETE `guest_tokens WHERE slot_id = slot.id` (cancel any pending invite).
7. Return `{ok: true}`.

The slot row itself stays — `auth_user_id` is preserved so historical `tournament_owner_id` rollups remain coherent; `claimed_at` is set to `NULL`.

### `guest-list-slots`

**Request:** `GET`, authenticated.

**Logic:**

1. Verify JWT → `user_id`. Reject if guest.
2. SELECT slots WHERE `parent_user_id = user_id` (server-derived; never trust client param). LEFT JOIN `profiles` on `auth_user_id` for the guest's display name + `active_sessions` for "currently online" status.
3. Return `[{slot_idx, auth_user_id, display_name, claimed_at, online}, ...]`.

### Guards on existing edge fns

- **`redeem-code`**: at top, after JWT verify, read `profiles.parent_user_id`. If non-null → 403 `guests cannot redeem codes`.
- **`game-create-session`**: same parent-guard, only when `mode IN ('live_room','tournament')`. Solo modes still allowed for guests. Then call `assertActiveSession`.
- **`game-submit-result`, `game-live-event`, `game-live-snapshot`**: call `assertActiveSession` after JWT verify.

## Risks and trade-offs

- **JWT residual window after revoke.** Even after `auth.admin.signOut('global')`, the access token remains valid until exp (default 1h). Mitigated by `assertActiveSession` checking `active_sessions` on every game-mutating call. The 1h window only matters for read-only fetches (e.g. `/figures` browse), which are acceptable for a casual game. Strong revoke is enforced where it matters: game state mutation.

- **Magic-link URL hygiene.** The token in `?token=` is convenient but leakable via Referer / browser history / screenshots. Mitigations: 15-min TTL, single-use (atomic delete), 32-byte entropy, `Referrer-Policy: no-referrer`, `history.replaceState` scrub on mount, `email_confirm: true` so the password-rotation step makes a leaked URL useless once claimed. Still, do not log full URLs anywhere — the `guest-generate-token` edge fn must not emit the raw token to logs.

- **Tournament fairness.** Treating parent + 5 guests as one tournament identity caps guest farming, but it also means a parent who shares slots with 5 friends only gets one shared submission — first to finish wins. This is the intended design (Codex review confirmed this is correct policy, not a bug). Surface this in the tournament rules screen so users aren't surprised.

- **Referer policy via `<meta>` tag.** `<meta name="referrer">` is honored by all evergreen browsers but is best-effort; the actual `Referrer-Policy` HTTP header would be ideal. Setting that on a Netlify-hosted SPA requires `_headers` config — included as a side change in this design (`public/_headers` adds `/guest/join Referrer-Policy: no-referrer`).

- **Parent thrashing / friend whiplash.** A malicious parent can repeatedly revoke + regenerate, kicking friends mid-game. Per-parent rate limit of 10 token generations/hour bounds this. Audit logs not in scope for v1; if abuse appears, add a `guest_audit` table.

- **Username partial-unique migration.** The starter dataset has no duplicate parent usernames (user has confirmed). The migration drops the global unique constraint and creates a partial unique index. If a future migration introduces a duplicate via service-role insert, the partial index will catch it; humans can never reach that state through normal sign-up flows.

- **`auth.admin.createSession` semantics.** Returning fresh `{access_token, refresh_token}` directly from the claim edge fn requires using the admin OTP-verification path (`generateLink` → `verifyOtp`) since Supabase doesn't expose a direct "issue tokens" admin API. This pattern is used by the existing `redeem-code` edge fn for code-based sign-in. Same approach reused here — no new attack surface.

- **`active_sessions` defense in depth retroactively fixes single-device sign-in.** The `assertActiveSession` helper plugs the access-token-after-revoke gap that exists today in the single-device feature. Shipping guest accounts therefore strengthens an existing feature without a separate migration. Worth calling out in the PR description.

- **Deno fetch / Edge Function rate-limit ceiling.** Supabase Edge Functions allow ~50 requests/sec/project. Our per-parent rate limit (10/hr) is way under any plausible throttle. Not a concern.

- **Card ownership for guests.** Guests intentionally own no cards. They can join rooms hosted by parents who own the relevant cards. This was Codex-flagged as worth confirming and is correct per the locked design pick #5.
